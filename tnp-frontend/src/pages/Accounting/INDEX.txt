# Accounting Frontend Codex Index

Updated: 2026-04-16

Use this file as the first read before editing the Accounting frontend. It is a compact map for `tnp-frontend/src/pages/Accounting` plus the shared RTK state under `tnp-frontend/src/features/Accounting`.

When adding or debugging frontend behavior, read `accountingApi.js` first, then the specific page folder. Only open backend files if the endpoint contract or DB flow is unclear.

## App Routes

All routes are nested under `/accounting` with `AccountingLayout` as the shell:

- `/accounting` -> `PricingIntegration/PricingIntegration.jsx`
- `/accounting/pricing-integration` -> `PricingIntegration/PricingIntegration.jsx`
- `/accounting/quotations` -> `Quotations/Quotations.jsx`
- `/accounting/quotations/report` -> `Quotations/report/QuotationReport.jsx`
- `/accounting/invoices` -> `Invoices/Invoices.jsx`
- `/accounting/delivery-notes` -> `DeliveryNotes/DeliveryNotes.jsx`

There is no dedicated top-level Receipts page in the current frontend. Receipt APIs are used by delivery-note source selection and invoice/payment flows.

Invoice deep links support `/accounting/invoices?selected=<invoiceId>`. `Invoices/Invoices.jsx` consumes `selected`, opens `InvoiceDetailDialog`, and removes the query param when the dialog closes.

## Domain Flow

The UI follows this document chain:

1. Completed Pricing Request
2. Quotation
3. Invoice
4. Receipt or tax/receipt PDF generated from invoice
5. Delivery Note

Pricing requests come from the Pricing subsystem, then Accounting creates quotations. Quotations can be approved, sent, completed, duplicated, converted to invoices, and synced to related invoices. Invoices support before/after deposit modes, evidence upload, approval, PDF output, and delivery-note source data. Delivery notes are created manually or from invoice/receipt sources and move through shipping/delivery/completion states.

## RTK Query Map

Main API file: `tnp-frontend/src/features/Accounting/accountingApi.js`

Base behavior:

- `reducerPath`: `accountingApi`
- `baseUrl`: `apiConfig.baseUrl`
- `credentials`: `apiConfig.credentials`
- `refetchOnFocus`: false
- `refetchOnReconnect`: true
- `keepUnusedDataFor`: 1800 seconds globally
- Tags: `PricingRequest`, `Quotation`, `Invoice`, `Receipt`, `DeliveryNote`, `Dashboard`, `Company`, `Customer`, `SyncJob`

- Quotations: `getQuotations`, `getQuotation`, `getQuotationDuplicateData`, `createQuotation`, `createQuotationFromPricing`, `createQuotationFromMultiplePricing`, `createStandaloneQuotation`, `updateQuotation`, `deleteQuotation`, `submitQuotation`, `approveQuotation`, `rejectQuotation`, `sendBackQuotation`, `revokeApprovalQuotation`, `markQuotationSent`, `markQuotationCompleted`, `sendQuotationEmail`

- Receipts: `getReceipts`, `getReceipt`, `createReceiptFromPayment`, `updateReceipt`, `approveReceipt`, `calculateVAT`, `uploadPaymentEvidence`, `generateReceiptPDF`
- Delivery notes: `getDeliveryNotes`, `getDeliveryNote`, `getDeliveryNoteInvoiceItems`, `getDeliveryNoteInvoices`, `createDeliveryNote`, `updateDeliveryNote`, `deleteDeliveryNote`, `createDeliveryNoteFromReceipt`, `startShipping`, `updateTracking`, `markDelivered`, `markCompleted`, `markFailed`, `getDeliveryTimeline`, `getCourierCompanies`, `getDeliveryMethods`, `generateDeliveryNotePDF`, `generateDeliveryNotePDFBundle`
- Utilities: `getStatuses`, `getDashboardStats`

Important cache notes:

- Pricing request autofill is cached long and bulk autofill serializes sorted PR ids to avoid duplicate fetches.
- Updating quotations invalidates quotation, invoice, dashboard, and sync tags because quotations can sync into related invoices.
- Invoice and delivery note mutations generally invalidate the entity tag plus list/dashboard tags.
- File uploads use `FormData` and often set `X-Skip-Json`.
- Receipt payloads are normalized in `accountingApi.js` before create/update/create-from-payment:
  - `receipt_type` becomes `type`
  - `vat_amount` becomes `tax_amount`
  - `payment_amount` becomes `total_amount`
  - `vat_type` is not sent to the backend
- Receipt VAT helper sends `amount` and `type`. It accepts `type`, `receiptType`, or a legacy `vatType` only if that value is already a valid receipt type.
- Receipt evidence upload sends multipart `files[]` plus optional `description`, and invalidates both receipt detail and list tags.

## Backend Endpoint Pairing

All URLs below are relative to `apiConfig.baseUrl`.

- `/pricing-requests*` -> `AutofillController`
- `/customers/search`, `/customers/{id}/details` -> `AutofillController`
- `/quotations*` -> `QuotationController`
- `/quotations/report*` -> `QuotationReportController`
- `/invoices*` -> `InvoiceController`
- `/receipts*` -> `ReceiptController`
- `/delivery-notes*` -> `DeliveryNoteController`
- `/companies*` -> backend Company controller, not under Accounting controller folder

Static backend routes must stay before wildcard routes, especially report routes before `/quotations/{id}`, invoice utility routes before `/invoices/{id}`, `/receipts/calculate-vat` before `/receipts/{id}`, and delivery utility routes before `/delivery-notes/{id}`.

Receipt backend contract notes:

- Current source of truth follows the legacy `receipts` table: `type`, `tax_amount`, `total_amount`, `payment_method`, `payment_reference`, `issued_by`, `approved_by`, `approved_at`, `created_at`.
- Backend still exposes compatibility aliases such as `receipt_type`, `vat_amount`, `payment_amount`, and `payment_date`.
- The current receipt table does not persist `payment_date`, `payment_amount`, `bank_name`, `vat_rate`, `vat_amount`, `created_by`, `updated_by`, `submitted_by`, or rejection/submission audit fields.
- Receipt evidence upload contract is `files[]`; do not send `evidence_file`.
- VAT contract is `amount` + `type`; do not send `vat_type`.

## Change Rules

- Add new Accounting server endpoints in `accountingApi.js`; export the generated hook at the bottom.
- Keep page files as composition shells; put orchestration in local hooks and pure transforms in local `utils`.
- Reuse `shared/components`, `components/SkeletonLoaders.jsx`, `utils/accountingToast.js`, and existing MUI patterns.
- For list mutations, confirm tag invalidation so list/detail/dashboard state refreshes without manual full-page reloads.
- For binary PDF downloads/previews, prefer the existing direct `fetch` pattern in `useInvoiceActions.js` or a targeted helper; RTK Query JSON assumptions can be awkward for blobs.
- Do not add a new global Redux field for server data already owned by RTK Query.
- When changing payload fields, update frontend adapter/form, `accountingApi.js`, backend Form Request, service method, model fillable/casts, and migrations/index docs together.
- When adding another invoice deep-link action, keep URL state and dialog close cleanup in `Invoices/Invoices.jsx` so browser navigation remains predictable.
